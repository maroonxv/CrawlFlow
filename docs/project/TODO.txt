当前开发重点如下：
1. 完善 `EventBus`。
2. 完善 `BaseEventHandler`。
3. 完善基于事件机制的日志系统，包括两部分：
   a. 通过 `logging_handler` 将日志写入日志文件。
   b. 通过 `websocket_handler` 将日志推送到浏览器前端。
4. 在 `test/unit` 中为上述组件补齐单元测试。
5. 在 `test/integration` 中编写整合事件与日志机制后的爬虫系统测试。


日志系统目标应覆盖以下五类日志：
1. 任务生命周期日志
   - 任务启动、停止、暂停、恢复
   - 任务配置（起始 URL、深度、策略）
   - 任务完成统计（总 URL 数、成功率、耗时）

2. URL 处理日志
   - URL 入队与出队
   - URL 去重命中与未命中
   - `robots.txt` 检查结果
   - HTTP 请求与响应（状态码、耗时）

3. 数据提取日志
   - PDF 链接提取（成功、失败、数量）
   - 元数据提取（标题、作者等字段提取情况）
   - 数据验证结果（字段缺失、格式错误）

4. 错误与异常日志
   - 网络异常（超时、连接失败、DNS 错误）
   - 解析失败（HTML 解析错误、编码问题）
   - 数据保存失败

5. 性能与监控日志
   - 速率控制触发情况（请求延迟、限流）
   - 队列状态（待爬取 URL 数量、队列深度）


项目采用领域驱动设计原则进行开发。
目前领域层与基础设施层已经完成，正在推进应用层。需要完成的任务包括：
1. 检查 `CrawlerService` 对领域层与基础设施层的调用是否正确。
2. 让 `CrawlerService` 作为应用层，将领域事件发布到总线上。
3. 在 `CrawlerService` 中增加一个查询某个 `CrawlTask` 最新 `results` 的方法。
   然后在 `_execute_crawl_loop` 的每一轮循环里，每当 `results` 新增一条结果时就查询一次，供 `crawler_view` 调用。


为了支撑以下业务能力，需要进一步梳理应用层（`CrawlerService`）与接口层（`crawler_view`）的编排方式：
- 前端浏览器可以逐个添加待爬取 URL。
- 可以设置请求间隔。
- 可以设置爬取策略（BFS、DFS、优先队列）。
- 可以控制爬取任务的开始、暂停与结束。

同时前端应支持实时查看：
- 当前任务的状态。
- 当前任务的实时日志。
- 当前任务的实时结果，并在每个 URL 爬取完成后即时刷新。


还需要使用 React.js 编写满足以下要求的前端页面：
1. 能够创建爬取任务。
2. 创建任务时能够填写所需参数。
3. 可以创建多个任务，并选择查看其中任意一个任务。
4. 同一时间只能有一个爬取任务处于运行状态，并且需要支持开始、暂停、继续与结束控制。
   当任务处于暂停状态时，只允许修改请求间隔、最大页面数与最大深度，其他参数不可修改。
5. 能够实时查看爬取任务日志，包括 `crawl_process`、`error`、`performance`、`task_lifecycle` 四类日志。
6. 爬取任务结束后，显示最终爬取结果。
